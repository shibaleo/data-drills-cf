# data-drills-cf フレームワーク選定 提案書

- 作成日: 2026-04-22
- 対象プロジェクト: data-drills-cf
- スコープ: Vite + React + Hono + Clerk 構成における追加フレームワーク選定
- ステータス: 意思決定前ドラフト

---

## 1. 目的

Vite + React + Hono + Clerk というスタックが定まった状態から、実装に入る前に「どのカテゴリで、何を、なぜ採用するか」を明示する。選定対象を最小に絞り、将来の入れ替えコストを下げることを優先する。

## 2. 前提

### 2.1 既定のスタック（変更しない）

| 層 | 採用 |
|---|---|
| ビルド・開発サーバー | Vite |
| UI フレームワーク | React 19 |
| ルーティング | TanStack Router |
| UI コンポーネント | shadcn/ui + Radix UI |
| スタイリング | Tailwind CSS v4 |
| API 層 | Hono (Cloudflare Workers) |
| 認証 | Clerk |
| ORM | Drizzle |
| DB | PostgreSQL (postgres.js 経由) |
| PDF/OCR サービス | 自前 API (Render) |
| サービス間認証 | Service Key |

### 2.2 設計原則

- **曖昧にしない**: フロントとバックの境界を明示的に保ち、プラットフォーム差し替えの自由度を保持する。
- **責任範囲を狭くする**: 各ライブラリが受け持つドメインを明確にし、越境しない。
- **後から足す**: 採用判断が不要なものは採用しない。必要になった箇所だけ追加する。

---

## 3. 状態管理の整理

「状態管理」という語が指す対象は4つに分解される。種類ごとに手段が違うため、単一ライブラリでの統一は推奨しない。

| 種類 | 具体例 | 手段 |
|---|---|---|
| ローカル UI 状態 | モーダル開閉、入力中文字 | `useState` / `useReducer` |
| グローバル UI 状態 | テーマ、サイドバー開閉 | 必要時のみ Zustand |
| サーバー状態 | API から取得したデータ | TanStack Query |
| URL 状態 | ページ番号、フィルター | TanStack Router |

**方針**: サーバー状態は専用ツールに分離し、URL 状態はルーターに持たせ、残りは React 組み込みで扱う。クライアント状態管理ライブラリは**初期採用しない**。

---

## 4. 追加選定対象

### 4.1 TanStack Query (採用)

- **役割**: サーバー状態のキャッシュ・再取得・ローディング/エラー状態の管理。
- **採用理由**:
  - Hono API の呼び出し結果を `useState` + `useEffect` で扱うと、キャッシュ・再検証・競合回避を都度書くことになり破綻する。
  - TanStack Router と同系統で、Devtools の一貫性も高い。
  - Suspense モードや楽観的更新など、将来の高度な UX にも拡張が効く。
- **代替検討**: SWR。機能はほぼ同等だが、TanStack エコシステムに揃える方が学習・保守コスト低。
- **責任範囲**: サーバー状態のみ。クライアント UI 状態は扱わない。

### 4.2 Zod (採用)

- **役割**: スキーマ定義と実行時バリデーション。
- **採用理由**:
  - フロント入力検証、Hono のリクエスト検証 (`@hono/zod-validator`)、Drizzle スキーマからの型導出 (`drizzle-zod`) を**同一スキーマで串刺し**できる。
  - 型の単一定義ソースになり、乖離が構造的に起こらない。
  - OpenAPI 生成 (`@hono/zod-openapi`) への移行パスがある。
- **責任範囲**: API 境界、フォーム、DB 入出力のバリデーション。

### 4.3 React Hook Form (採用、ただし導入タイミングは可変)

- **役割**: フォームの状態・バリデーション・送信制御。
- **採用理由**:
  - 問題作成・編集・学習記録入力など複数フィールドの UI が増える見込み。
  - 非制御コンポーネント設計で再レンダリングが少なく、大きなフォームでも軽量。
  - Zod との統合 (`@hookform/resolvers/zod`) が成熟している。
- **代替検討**: TanStack Form。設計は現代的だが現時点で成熟度・事例量で React Hook Form に劣る。
- **導入タイミング**: 最初のフォームを書く時点。それまでは不要。

### 4.4 API 型共有方針: Hono RPC (採用)

- **役割**: Hono のルート型定義をフロントで直接参照し、型安全なクライアントを得る。
- **採用理由**:
  - 追加依存ゼロ (Hono 本体に含まれる)。
  - tRPC 的な型体験が、フレームワークロックインなしで得られる。
  - 個人利用・内部利用に閉じている現段階では、OpenAPI 生成のオーバーヘッドは過剰。
- **将来の切り替え**: 外部クライアント (税務顧問先向け SaaS、MCPist モジュール等) に API を公開する時点で、`@hono/zod-openapi` ベースの OpenAPI 公開に移行する。その際も Zod スキーマは再利用できる。

## 5. 見送り・保留項目

| 項目 | 判断 | 理由 |
|---|---|---|
| Zustand / Jotai | **初期採用しない** | サーバー状態は TanStack Query、URL 状態は Router、残りは `useState` で足りる見込み。複数コンポーネント共有のローカル UI 状態が実際に発生した時点で再検討。 |
| Redux / Redux Toolkit | **採用しない** | 現代的な分離構成ではスコープが過大。 |
| tRPC | **採用しない** | Hono RPC で同等の体験が得られるため重複。 |
| テストフレームワーク | **保留** | 初期実装後、リグレッション防止が必要になった段階で Vitest + Playwright を導入。 |
| i18n ライブラリ | **保留** | 現時点で単一言語。多言語要件が出た時点で `react-i18next` を検討。 |
| Sentry 等エラー監視 | **本番運用開始時に導入** | 開発中は Cloudflare / Render のログで十分。 |

---

## 6. 導入順序

1. **Zod** を最初に入れる。これが全てのスキーマの土台になる。
2. **TanStack Query** を Hono API 呼び出しの最初のタイミングで導入。QueryClient の Provider を `main.tsx` 付近に配置。
3. **Hono RPC クライアント** を立ち上げ、`hc<AppType>` ベースの API ラッパーを作成。TanStack Query の `queryFn` から呼ぶ構造にする。
4. **React Hook Form + zodResolver** を、最初のフォーム実装時点で導入。
5. それ以降は必要に応じて追加判断。状態管理ライブラリは原則として入れない。

---

## 7. リスク・トレードオフ

| リスク | 対処 |
|---|---|
| Hono RPC の型推論が巨大プロジェクトでコンパイル時間を増やす | 将来問題化した場合は Hono のルートを複数アプリに分割し、型境界を切る。 |
| TanStack Query の設定差 (staleTime, gcTime 等) がチーム内で揺れる | プロジェクト既定値を `queryClient` 初期化時に一元設定し、個別上書きは例外扱いとする。 |
| Zod のバンドルサイズ (約 14KB gzip) | 現時点の SPA 規模では許容。Zod v4 または valibot への移行余地を残す。 |
| React Hook Form 未導入で素の state のまま書き続けると負債化 | 2 フィールド以上のフォームは最初から React Hook Form を使う規律。 |

---

## 8. 意思決定の確定内容

- **採用**: TanStack Query / Zod / React Hook Form / Hono RPC
- **初期見送り**: クライアント状態管理ライブラリ / テスト / i18n / 監視
- **変更しない**: 既定スタック (Vite / TanStack Router / shadcn/ui / Tailwind v4 / Hono / Clerk / Drizzle / PostgreSQL)

以上の方針に基づき、実装フェーズに入る。
